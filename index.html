Linux File System Hierarchy

A structured guide to the FSH standard directories

Introduction to Linux File System

  • Central Concept: Everything in Linux starts from a single point: the / root directory.
  • Unified File System: Unlike Windows (which uses separate drive letters for partitions), Linux has a single, unified file system hierarchy where all files and directories branch out from the root. External drives and partitions are mounted as subdirectories within this hierarchy.
  • Organization: Top-level directories are grouped into categories for clarity, separating system components, user files, temporary data, and applications.

Key Directories Explained

1. / (Root Directory)

  • Definition: The absolute top-level directory in the Linux file system. All other directories and files descend from it.
  • Purpose: The starting point for the entire system's file structure.
  • Distinction: Not to be confused with /root.

2. /root

  • Definition: The home directory for the root user (administrator).
  • Purpose: Keeps the root user's personal files and configurations isolated and secure from regular user data.
  • Location: It is a subdirectory of /, not /home.

3. /bin (Binaries)

  • Definition: Contains essential command-line tools (binaries) required for the system to function, especially during boot or in recovery mode.
  • Accessibility: Commands here are always available to all users.
  • Criticality: Crucial for everyday use and troubleshooting.
Caution: Modifying this directory can break core system functionality.

4. /sbin (System Binaries)

  • Definition: Holds specialized system utilities primarily used by the system administrator (root user).
  • Privileges: Most commands require superuser privileges to run.
  • Purpose: Critical for managing and maintaining the system.
Caution: Do not modify without understanding the risks.

5. /lib (Libraries)

  • Definition: Stores shared libraries that support the commands found in /bin and /sbin. Also contains kernel modules.
  • Function: Provides common functionality (e.g., file reading, I/O) to avoid duplicating code in individual commands.
  • Variations: May include /lib32 and /lib64 for 32-bit and 64-bit libraries.
  • Kernel Modules: Small pieces of code loaded into the Linux kernel on demand to extend its functionality.
Caution: Tampering can prevent the system from booting or break basic commands.

6. /usr (Unix System Resources)

  • Definition: Contains most of the system's userspace applications, libraries, and documentation. Despite the name, it does not store personal user files.
  • Structure: Mirrors several root-level directories (e.g., /usr/bin, /usr/lib, /usr/sbin) for non-essential software.
  • Purpose: Keeps the system organized and bootable even if only the root partition is available.
  • Management: Usually read-only and managed by the system.

7. /boot

  • Definition: Contains all the files required for the system to boot up.
  • Key Contents:
    • Linux kernel
    • Initial RAM disk (initrd or initramfs)
    • GRUB bootloader files (grub.cfg)
Caution: Even small changes can render the system unbootable. Often placed on a separate partition.

8. /dev (Devices)

  • Definition: A virtual file system containing device files that act as interfaces to hardware. It does not hold regular files stored on disk.
  • Concept: In Linux, everything is treated like a file, including hardware devices.
  • Types:
    • Block devices: Handle data in chunks (e.g., hard drives).
    • Character devices: Handle data as a stream of characters (e.g., keyboards).
  • Dynamic Management: Linux adds or removes these files as hardware changes.
Caution: Interacting with these files directly can affect hardware.

9. /etc (Et Cetera / Editable Text Configuration)

  • Definition: The central location for system-wide configuration files.
  • Format: Most are plain text files, readable and editable by system administrators.
  • Purpose: Defines how the system behaves, covering network settings, user accounts, startup scripts, and service configurations.
Caution: Changes must be made carefully as they affect the entire system.

10. /home

  • Definition: The personal space for regular users. Each user has their own subdirectory here.
  • Purpose: Stores user documents, downloads, projects, and personal settings, keeping user data separate from system files.
  • User Configuration: Contains user-specific configuration files, often hidden (starting with a dot, e.g., .bashrc).
  • Permissions: Users typically don't have access to each other's home folders by default.
  • Customization: Designed to be customized by users.

11. /media

  • Definition: Where modern Linux systems automatically mount removable media.
  • Purpose: Provides user-friendly access to the contents of devices like USB drives, CDs, and DVDs.
  • Automation: Often handled seamlessly by desktop environments.
  • Distinction: Contrasts with /dev (raw device connection) and /mnt (manual mounts).

12. /mnt (Mount)

  • Definition: Traditionally used for manually mounting external storage.
  • Purpose: Reserved for temporary mounts by system administrators for tasks like accessing failing hard drives or network shares during troubleshooting.
  • Contents: Empty by default, only contains content when something is actively mounted.
  • Usage: Typically used by the root user or advanced scripts, not for automatic user access.

13. /proc (Processes)

  • Definition: A virtual file system (like /dev) that provides a dynamic, in-memory view of the Linux kernel and running processes.
  • Key Contents:
    • Folders named after Process IDs (PIDs) with details about each process.
    • System information files (e.g., /proc/cpuinfo, /proc/meminfo).
  • Purpose: Vital for monitoring, debugging, and interacting with the system's inner workings in real-time.

14. /sys (System)

  • Definition: A virtual directory (SYSFS pseudo file system) that acts as a dynamic interface between the Linux kernel and hardware.
  • Purpose: Exposes information about devices, drivers, and system buses. Allows advanced users or tools to tweak kernel settings on the fly.
  • Structure: Hierarchical, representing each hardware component and subsystem.
  • Backend: Key backend for system utilities like udev and power managers.

15. /run

  • Definition: An in-memory directory holding temporary runtime data used by the system and applications right after boot.
  • Persistence: Created early in startup, everything inside disappears on reboot.
  • Purpose: Used by running processes for communication, state tracking, and storing short-lived files (e.g., process IDs, sockets).
  • Replacement: Replaced older directories like /var/run and /var/lock.

16. /srv (Service)

  • Definition: Where data for services provided by the system is stored.
  • Purpose: Holds the content served to users connecting to the machine remotely (e.g., web server files, FTP data).
  • Usage: More important on multi-user or enterprise server setups. Not all Linux systems use it by default.

17. /var (Variable)

  • Definition: Stores files that change frequently during system operation.
  • Key Contents:
    • Logs (/var/log)
    • Caches
    • Spool files
    • Package manager metadata (e.g., /var/lib for Debian-based systems)
  • Security: Can contain sensitive information and should be properly protected.
  • Management: Often placed on its own partition in server setups to prevent filling up the root file system.

18. /tmp (Temporary)

  • Definition: Stores temporary files created by applications or users.
  • Persistence: Files are typically deleted automatically on reboot or after a certain period.
  • Permissions: Writable by all users, but the sticky bit ensures users can only delete files they own.
  • Location: On some systems, it resides in memory (tmpfs), so writing large files can consume RAM.

19. /opt (Optional)

  • Definition: Where optional or third-party software is installed.
  • Organization: Each application usually has its own subdirectory (e.g., /opt/google/chrome).
  • Purpose: Provides a safe space for software that doesn't follow standard Linux file structure rules, without interfering with core system files. Common in enterprise environments.
  • Privileges: Modifying requires administrative privileges.

Actionable Steps

  1. Understand Before Modifying: Never modify core system directories like /bin, /sbin, /lib, /boot, /dev unless you know exactly what you're doing and are following specific instructions. Incorrect changes can render your system unbootable or unstable.
  2. Customize Safely: The /home directory is your personal space; feel free to customize and save files here.
  3. Troubleshooting: When troubleshooting, examine /var/log for system logs, /etc for configuration issues, and /proc or /sys for real-time system and hardware information.
  4. Disk Management: Be aware of /var and /tmp for disk space management, especially on servers. Consider placing /var on a separate partition to prevent the root file system from filling up.
  5. Access Control: Remember that commands in /sbin typically require sudo or root privileges.
